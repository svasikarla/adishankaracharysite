## Component Architecture

### Architecture Philosophy
This project follows the **shadcn/ui** component composition pattern:
- Components are built on **Radix UI primitives** (headless, accessible components)
- Small, reusable components composed to build larger features
- Component variants managed via **class-variance-authority** (CVA)
- Server Components by default, Client Components only when needed (`"use client"`)

### Component Categories

#### 1. **Primitive UI Components** (50 components in `components/ui/`)
All components follow this pattern:
- Built on Radix UI primitives
- Styled with Tailwind CSS
- Exported as named exports
- Use `forwardRef` for DOM access
- Include `displayName` for debugging

**Example Component Pattern:**
```typescript
"use client" // Only if client-side functionality needed

import * as React from "react"
import { forwardRef } from "react"
import { cn } from "@/lib/utils"

interface ComponentProps extends React.HTMLAttributes<HTMLDivElement> {
  variant?: 'default' | 'outline'
}

const Component = forwardRef<HTMLDivElement, ComponentProps>(
  ({ className, variant = 'default', ...props }, ref) => (
    <div
      ref={ref}
      className={cn('base-classes', className)}
      {...props}
    />
  )
)
Component.displayName = 'Component'

export { Component }
```

#### 2. **Custom Components**

**DarkModeToggle** (`components/dark-mode-toggle.tsx`)
- **Type:** Client Component (`"use client"`)
- **Purpose:** Toggle between light/dark mode
- **State Management:** React useState + localStorage
- **Implementation:**
  - Persists theme in `localStorage.theme`
  - Respects system preference if no stored value
  - Toggles `dark` class on `document.documentElement`
  - Uses Lucide icons (SunIcon/MoonIcon)
- **Note:** Custom implementation instead of using next-themes provider

**Header** (`components/Header.tsx`)
- **Type:** Server Component (default)
- **Purpose:** Site header with logo and navigation
- **Unique Approach:** Uses inline styles instead of Tailwind for some styling
- **Features:**
  - Next.js Image optimization
  - Responsive sizing
  - Placeholder logo

**ThemeProvider** (`components/theme-provider.tsx`)
- **Type:** Client Component wrapper
- **Purpose:** Wraps next-themes provider for app-wide theme support
- **Note:** Currently not used in app/layout.tsx (dark mode handled by custom toggle)

### Component Naming Conventions

- **File Names:**
  - Components: PascalCase (`Button.tsx`, `DarkModeToggle.tsx`)
  - Hooks: kebab-case with `use-` prefix (`use-mobile.tsx`, `use-toast.ts`)
  - Utilities: kebab-case (`utils.ts`)

- **Exports:**
  - Named exports (not default exports)
  - Compound components exported together (e.g., `Card`, `CardHeader`, `CardTitle`)
  - Variant functions exported alongside components (e.g., `buttonVariants`)

### Compound Component Pattern

Many components use compound patterns for better composition:

```typescript
// Card component with subcomponents

<Card>
```

## Key Conventions

### File and Import Conventions

#### Path Aliases (configured in `tsconfig.json`)
Always prefer path aliases over relative imports:

```typescript
// GOOD
import { Button } from "@/components/ui/button"
import { cn } from "@/lib/utils"
import { useMobile } from "@/hooks/use-mobile"

// BAD
import { Button } from "../../components/ui/button"
import { cn } from "../lib/utils"
```

**Configured Aliases:**
- `@/*` → Root directory
- `@/components/*` → Components directory
- `@/lib/*` → Library utilities
- `@/hooks/*` → Custom hooks
- `@/ui/*` → UI components

#### Import Order Convention
1. React and Next.js imports
2. Third-party libraries
3. Radix UI components
4. Internal components (via `@/`)
5. Utilities and helpers
6. Types and interfaces

### TypeScript Conventions

#### Type Safety
- **Strict Mode:** Enabled in `tsconfig.json`
- **Explicit Props:** Always define props interfaces
- **Extend HTML Attributes:** Use `React.HTMLAttributes<T>` for extensibility
- **Ref Forwarding:** Use `forwardRef` for components that expose DOM elements

```typescript
// Component props pattern
interface ButtonProps extends React.ButtonHTMLAttributes<HTMLButtonElement> {
  variant?: 'default' | 'outline' | 'ghost'
  size?: 'default' | 'sm' | 'lg' | 'icon'
  asChild?: boolean
}

const Button = forwardRef<HTMLButtonElement, ButtonProps>(
  ({ className, variant = 'default', size = 'default', ...props }, ref) => {
    // Component implementation
  }
)
Button.displayName = 'Button'
```

### Component Conventions

#### Client vs Server Components
- **Default:** Server Components (no directive needed)
- **Client Components:** Add `"use client"` directive at top of file
- **When to use client components:**
  - State management (useState, useReducer)
  - Effects (useEffect, useLayoutEffect)
  - Browser APIs (localStorage, window, document)
  - Event handlers (onClick, onChange)
  - Custom hooks that use client features

#### Display Names
Always set `displayName` for debugging:
```typescript
Component.displayName = 'Component'
```

### Dark Mode Convention

#### Implementation Details
- **Strategy:** Class-based (`.dark` class on `<html>` element)
- **Persistence:** `localStorage.theme` stores user preference
- **System Preference:** Falls back to `prefers-color-scheme: dark` if no stored value
- **Toggle Location:** Custom component at `components/dark-mode-toggle.tsx`

#### Dark Mode Classes Pattern
```typescript

  )
```

## Important Notes for AI Assistants

### When Making Changes

#### DO:
1. **Preserve Existing Patterns:** Follow the established component and styling patterns
2. **Use Path Aliases:** Always use `@/` imports instead of relative paths
3. **Maintain Type Safety:** Add proper TypeScript types for all new code
4. **Respect Dark Mode:** Always provide both light and dark variants for new styles
5. **Use shadcn/ui:** Leverage existing UI components before creating custom ones
6. **Follow File Naming:** Use PascalCase for components, kebab-case for utilities/hooks
7. **Set displayName:** For all new components using forwardRef
8. **Optimize Images:** Use Next.js Image component for all images
9. **Prefer Tailwind:** Use utility classes over inline styles
10. **Test Both Modes:** Verify changes in both light and dark mode

#### DON'T:
1. **Mix Styling Approaches:** Don't mix CSS modules, styled-components, or vanilla CSS
2. **Use Default Exports:** Components use named exports
3. **Ignore Accessibility:** Radix UI provides accessibility; don't remove it
4. **Hardcode Colors:** Use CSS variables or Tailwind theme colors when possible
5. **Create New Build Config:** Don't modify Next.js, Tailwind, or TypeScript configs without reason
6. **Add Unnecessary Dependencies:** Check if shadcn/ui or Radix UI already provides the functionality
7. **Skip Mobile Responsiveness:** Always include responsive breakpoints
8. **Remove forwardRef:** Many components need ref forwarding for proper functionality
9. **Ignore ESLint:** Run `npm run lint` before committing
10. **Use Client Components Unnecessarily:** Default to Server Components; only use "use client" when needed

- CSS variables in `app/globals.css` (using HSL format)
